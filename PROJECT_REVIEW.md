## 1. 기술적 실현성 (75/100)

### ✅ 강점

1. **검증된 기술 스택 사용**
   - librosa, Demucs, yt-dlp 모두 성숙한 오픈소스
   - 아까 실제로 작동하는 코드 시연함
   - Streamlit으로 빠른 프로토타이핑 가능

2. **분류 로직이 논리적**
   - 보컬 MBTI 4축 설계가 음악적으로 타당
   - 특징 추출 → 분류 → 해석 파이프라인 명확

3. **보컬 분리 기술 대안 확보**
   - Demucs, Spleeter, LALAL.AI 등 폴백 옵션 있음

### ❌ 약점 및 리스크

1. **음정 분석의 근본적 한계** ⚠️ CRITICAL
   ```
   문제: librosa.pyin()은 "가장 강한 피치"만 추출
   실제 상황: 반주, 하모니, 리버브가 섞인 환경
   결과: 분석 정확도가 크게 떨어질 수 있음
   ```
   
   **해결 제안**:
   - 반드시 보컬 분리 전처리 필수화
   - 분리 품질 신뢰도를 결과에 명시
   - "분석 신뢰도 70% - 솔로 녹음 권장" 같은 경고

2. **MBTI 분류 검증 부재** ⚠️ HIGH
   ```
   문제: 분류 기준값(threshold)이 경험적 추정
   질문: "spectral_centroid < 1800Hz면 따뜻한 톤"의 근거는?
   리스크: 잘못된 분류 → 잘못된 추천 → 신뢰도 하락
   ```
   
   **해결 제안**:
   - 실제 찬양 인도자 50명+ 샘플로 캘리브레이션
   - A/B 테스트로 분류 정확도 검증
   - "베타 버전" 명시하고 피드백 수집

3. **LLM 의존성 미정의**
   ```
   문제: 감성 해석에 LLM을 쓴다고 했지만, API 비용/속도 고려 안됨
   현재: 템플릿 기반 로컬 해석만 구현
   ```
   
   **해결 제안**:
   - 기본은 템플릿, 프리미엄은 LLM으로 티어 분리
   - LLM 호출 캐싱으로 비용 절감

4. **처리 시간 문제**
   ```
   - 유튜브 다운로드: 30초~1분
   - 보컬 분리 (Demucs): 2~5분
   - 분석: 30초~1분
   총: 3~7분 대기 → 이탈률 높음
   ```
   
   **해결 제안**:
   - 비동기 처리 + 이메일/푸시 알림
   - 프로그레스 바에 "현재 단계" 상세 표시
   - 빠른 미리보기 → 상세 분석 2단계

## 2. 비즈니스 가치 (62/100)

### ✅ 강점

1. **명확한 니치 마켓**
   - 한국 교회 찬양팀 타겟 (약 10만+ 팀 추정)
   - 기존 보컬 앱에 없는 "영성" 요소

2. **바이럴 포인트**
   - "보컬 MBTI" → SNS 공유 유도
   - "당신은 김윤진 스타일!" → 공감 유발

### ❌ 약점 및 리스크

1. **TAM(시장 규모) 불확실** ⚠️ HIGH
   ```
   질문: 실제로 돈을 낼 사용자가 몇 명?
   
   현실:
   - 무료 앱에 익숙한 교회 문화
   - 찬양팀 예산은 보통 제한적
   - 개인 결제 vs 교회 단체 결제?
   ```
   
   **필요한 검증**:
   - 10명의 찬양 인도자 인터뷰
   - "월 9,900원 내겠습니까?" 직접 물어보기
   - 무료 사용자 중 전환율 추정

2. **경쟁사 대응 리스크**
   ```
   Smule, Yousician 같은 대형 앱이
   "워십 모드" 추가하면?
   
   현재 해자(moat):
   - 한국 CCM 특화 데이터
   - 롤모델 DB (어노인팅, 마커스 등)
   - 영성 언어 번역
   
   하지만 이건 쉽게 복제 가능
   ```
   
   **해결 제안**:
   - 커뮤니티 구축 (찬양팀 네트워크)
   - 독점 파트너십 (어노인팅, 마커스와 협업)
   - 데이터 축적 (분석 결과 → 추천 정확도 향상)

3. **수익 모델 불명확**
   ```
   제안된 모델: 
   - Free: 3곡/월
   - Pro: ₩9,900/월
   
   문제:
   - 무료로 3곡이면 대부분 충분
   - 프로 전환 동기 부족
   ```
   
   **대안 제안**:
   - 무료: 기본 분석만 (MBTI + 점수)
   - 프로: 상세 구간 분석 + 연습 처방 + 성장 추적
   - 팀: 찬양팀 전체 분석 + 파트 배치 추천

## 3. 사용자 경험 (70/100)

### ✅ 강점

1. **감성적 언어 번역**
   - "20 cents 오차" → "자연스럽게 흔들리는" 
   - 이건 진짜 좋은 UX 포인트

2. **롤모델 매칭**
   - "당신은 김윤진 스타일" → 동기부여 효과

3. **구체적 연습 처방**
   - "립 트릴 30초" → 실행 가능한 액션

### ❌ 약점 및 리스크

1. **온보딩 복잡도** ⚠️ MEDIUM
   ```
   현재 플로우:
   1. URL 입력 또는 파일 업로드
   2. 시간 구간 지정
   3. 녹음 환경 선택
   4. 분석 대기 (3-7분)
   5. 결과 확인
   
   문제: 첫 사용까지 너무 많은 단계
   ```
   
   **해결 제안**:
   - 파일 업로드만으로 원클릭 분석 (기본 설정 자동)
   - 유튜브 연동은 "고급 옵션"으로

2. **결과 해석의 주관성**
   ```
   "스토리텔러 타입"이라고 했는데...
   사용자: "나는 파워풀하다고 생각하는데?"
   
   MBTI 결과에 동의하지 않으면 신뢰도 하락
   ```
   
   **해결 제안**:
   - "당신은 어떤 타입이라고 생각하세요?" 사전 설문
   - 불일치 시: "의외의 결과네요! 이런 면도 있어요"
   - 여러 곡 분석 후 종합 판단

3. **재방문 동기 부족**
   ```
   한 번 분석하고 끝?
   
   필요: 성장 추적, 연습 기록, 비교 기능
   현재: 구현 안됨
   ```

1. **저작권 리스크** 🔴 CRITICAL
   ```
   문제: YouTube 오디오 추출은 법적으로 회색 영역
   
   YouTube ToS:
   "다운로드 금지, 단 YouTube가 제공하는 경우 제외"
   
   리스크:
   - Google의 법적 조치 가능성
   - 앱스토어 거부 사유
   ```
   
   **해결 방안**:
   - 사용자 본인 녹음 업로드 위주로 유도
   - YouTube 연동은 "학습/연습 목적" 면책 조항
   - 추출 파일 저장하지 않고 즉시 분석 후 삭제

2. **개인정보 리스크** 🟡 MEDIUM
   ```
   - 음성 데이터는 생체정보
   - GDPR/개인정보보호법 적용 가능
   ```
   
   **해결 방안**:
   - 서버에 음성 저장 안함 (로컬 처리)
   - 또는 명시적 동의 + 삭제 옵션

3. **기술 부채** 🟡 MEDIUM
   ```
   - Streamlit은 프로토타입용, 프로덕션 한계
   - 확장성 고려 안됨
   ```
   
   **해결 방안**:
   - MVP 검증 후 FastAPI + React로 재구축
   - 처음부터 API 분리 설계
